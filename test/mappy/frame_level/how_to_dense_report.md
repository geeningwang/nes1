# How to Generate the Dense Frame Comparison Report

This document describes the full pipeline for producing
`test/mappy_out/dense_report.html` — a frame-by-frame pixel and
CPU/PPU state comparison between **nes1** and **FCEUX** for the
Mappy (Japan) ROM.

---

## Prerequisites

| Item | Path / version |
|---|---|
| Visual Studio 2022 | with C++ Desktop workload |
| FCEUX 2.6.6 (64-bit) | `fceux64.exe` on PATH, or adjust paths below |
| PowerShell 5.1 | built into Windows (`C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe`) |
| ROM | `test\Mappy (Japan).nes` |

---

## Step 1 — Build nes1.exe

Open `src\nes1.sln` in Visual Studio and build the **x64 Debug**
configuration, or from a Developer Command Prompt:

```powershell
& 'C:\Program Files\Microsoft Visual Studio\2022\Community\MSBuild\Current\Bin\MSBuild.exe' `
    'C:\Work\nes1\src\nes1.sln' /p:Configuration=Debug /p:Platform=x64 /t:Rebuild /v:m
```

Output binary: `src\x64\Debug\nes1.exe`

---

## Step 2 — Generate FCEUX reference output

Run the dense Lua script under FCEUX. This saves 270 BMP screenshots
to `test\mappy_out\fceux_dense_out\`.

```
fceux64.exe -lua "C:\Work\nes1\test\mappy_fceux_dense.lua" "C:\Work\nes1\test\Mappy (Japan).nes"
```

> **Note:** The Lua script calls `emu.exit()` automatically after
> frame 270. BMP output files are named `fceux_frame_NNNN.bmp`
> (NNNN = 0001–0270).

The matching `.txt` state files (`fceux_frame_NNNN.txt`) are **not**
produced by the Lua script. They are generated by running nes1 in
`--frametest` mode against the same ROM with a known-good reference
build, then renaming the outputs:

```powershell
# Run with reference build -> nes1_dense_ref\ (temp dir)
& 'C:\Work\nes1\src\x64\Debug\nes1.exe' `
    'C:\Work\nes1\test\Mappy (Japan).nes' `
    --frametest 'C:\Work\nes1\test\mappy_out\nes1_dense_ref' 270 1

# Rename nes1_frame_NNNN.txt -> fceux_frame_NNNN.txt
Get-ChildItem 'C:\Work\nes1\test\mappy_out\nes1_dense_ref\*.txt' |
    ForEach-Object {
        $dst = Join-Path 'C:\Work\nes1\test\mappy_out\fceux_dense_out' `
                         ($_.Name -replace '^nes1_', 'fceux_')
        Copy-Item $_.FullName $dst
    }
```

**The reference BMP and txt files need to be regenerated when:**
- FCEUX reference BMPs change (run the Lua step above)
- The txt format changes (e.g., new sections added to `export_frame()`)

When regenerating txt files, use a build that is known correct for the
frames you want as reference.

### What the txt file contains

Each `fceux_frame_NNNN.txt` is produced by nes1's own
`ppu_2c02::export_frame()` function. The format is identical to nes1's
output and includes a CPU RAM section (`=== CPU RAM ($0000-$07FF) ===`,
64 rows of 32 hex bytes) appended after the ASCII Screen section.

```
=== CPU State ===
PC=C06D A=00 X=5D Y=00 S=5F P=27 cycles=...

=== PPU Registers ===
PPUCTRL  ($2000): 88  (...)
PPUMASK  ($2001): 1E  (...)
PPUSTATUS($2002): 00  (...)
v=XXXX  t=XXXX  fine_x=N  mirroring=vertical
render_scroll_x=... render_scroll_y=... render_scroll_nt=...

=== Nametable ($2000-$23BF, 32x30 tile IDs) ===
...30 rows of 32 hex tile IDs...

=== Palette ($3F00-$3F1F) ===
BG:  ...
SPR: ...

=== Active Sprites (OAM) ===
  # |  Y   X  Tile Attr
  ...

=== ASCII Screen (32x30 tiles) ===
...

=== CPU RAM ($0000-$07FF) ===
...64 rows of 32 hex bytes (2048 bytes total)...
```

---

## Step 3 — Generate nes1 output

Run nes1 in `--frametest` mode. This saves 270 `.bmp` and `.txt` files
(one per frame, interval=1) to `test\mappy_out\nes1_dense_out\`.

```powershell
& 'C:\Work\nes1\src\x64\Debug\nes1.exe' `
    'C:\Work\nes1\test\Mappy (Japan).nes' `
    --frametest 'C:\Work\nes1\test\mappy_out\nes1_dense_out' 270 1
```

Output files are named `nes1_frame_NNNN.bmp` and `nes1_frame_NNNN.txt`.

**This step must be repeated every time nes1.exe is rebuilt.**

---

## Step 4 — Generate the dense HTML report

```powershell
powershell.exe -ExecutionPolicy Bypass `
    -File 'C:\Work\nes1\test\make_dense_report.ps1'
```

The script compares all 270 frame pairs and writes
`test\mappy_out\dense_report.html`.

Default parameters (can be overridden):

| Parameter | Default |
|---|---|
| `-Nes1Dir` | `C:\Work\nes1\test\mappy_out\nes1_dense_out` |
| `-FceuxDir` | `C:\Work\nes1\test\mappy_out\fceux_dense_out` |
| `-OutHtml` | `C:\Work\nes1\test\mappy_out\dense_report.html` |
| `-NFrames` | `270` |

### What the report compares, per frame

| Column | Fail criterion |
|---|---|
| **Pixels** | Any pixel differs between NES1 and FCEUX BMP (tolerance = 0) |
| **CPU** | A / X / Y / S / P registers differ. PC and cycle count are *shown as info only* (idle-loop capture timing differs) |
| **PPUCTRL / PPUMASK** | Register value differs |
| **PPUSTATUS** | *Info only* — spr0-hit and sprite-overflow flags are timing-dependent |
| **Palette ($3F00-$3F1F)** | Any palette entry differs (mirror slots $3F10/$3F14/$3F18/$3F1C are normalised before comparison) |
| **Nametable** | Any tile byte differs |
| **OAM** | Any active sprite entry differs |
| **RAM ($0000-$07FF)** | Any byte in the 2 KB CPU work-RAM differs (shown as row-diff count, 32 bytes/row). Shown as *N/A* if the txt files predate RAM export support |
| **V / T registers** | *Info only* — frame-end V diverges architecturally without causing a pixel error |
| **render\_scroll** | *Info only* — derived from V |

A frame is **PASS** only if pixels, CPU (A/X/Y/S/P), PPUCTRL/PPUMASK,
Palette, Nametable, OAM, and RAM all match. RAM is only checked when
both txt files contain a `=== CPU RAM` section (produced by nes1 builds
that include RAM export; older txt files show *N/A* and are not failed).

---

## Interpreting the report

Open `dense_report.html` in any browser. Each row is one frame.

- **Green rows** = PASS (all compared fields match)
- **Red rows** = FAIL (at least one compared field differs)
- Click a thumbnail to zoom in to full size
- The **V/T (info)** column is highlighted in orange/teal when V
  differs but is *not* a failure criterion (V diverges architecturally
  at frame-end)

## Quick re-run after a code change

```powershell
# 1. Rebuild
& 'C:\Program Files\Microsoft Visual Studio\2022\Community\MSBuild\Current\Bin\MSBuild.exe' `
    'C:\Work\nes1\src\nes1.sln' /p:Configuration=Debug /p:Platform=x64 /v:m

# 2. Re-run nes1 frametest
& 'C:\Work\nes1\src\x64\Debug\nes1.exe' `
    'C:\Work\nes1\test\Mappy (Japan).nes' `
    --frametest 'C:\Work\nes1\test\mappy_out\nes1_dense_out' 270 1

# 3. Regenerate report
powershell.exe -ExecutionPolicy Bypass `
    -File 'C:\Work\nes1\test\make_dense_report.ps1'

# 4. Open report
Start-Process 'C:\Work\nes1\test\mappy_out\dense_report.html'
```

## Additional test modes

### Scanline trace (per-scanline CPU+PPU state for one frame)

```powershell
& 'C:\Work\nes1\src\x64\Debug\nes1.exe' `
    'C:\Work\nes1\test\Mappy (Japan).nes' `
    --scanlinetrace 'C:\Work\nes1\test\mappy_out\nes1_sl_trace_fNNN.txt' NNN
```

Outputs one line per visible scanline (0–239) with:
`SL  CPU_CYC  PC=... A=... X=... Y=... S=... P=...  V=... T=... fx=N w=N  x=N y=N nt=N  P3F00`

Useful for diagnosing scroll register divergence at specific scanlines.

### FCEUX scanline trace

Use `test\mappy_ppu_trace.lua` under FCEUX to generate an equivalent
per-scanline trace from the reference emulator.
